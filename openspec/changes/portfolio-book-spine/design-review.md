# Design Review — `portfolio-book-spine` (adversarial, code-grounded)

**Reviewer:** Software Architect lens (Opus 4.8). Source code as primary evidence; the
proposal summary treated as a claim to verify. Out of scope per instructions:
`candidates/`, `researched/` (strategies will be rebuilt).

---

## VERDICT: **sound-with-corrections**

The root-cause diagnosis is correct and verified against source. The *unit is the trade*:
the scored statistic is a linear per-trade sum (`engine/evaluation.py:108-111`), the only
exposure-aware surface is a fail-open side-channel reconstructed from the trade bag
(`core/portfolio_foundation.py:642`, `_build_portfolio_foundation` `runner/__init__.py:852`),
funding/PnL is modeled three times, and the codebase *concedes* the defect
(`validation/agreement.py:94-97`). The proposed primitive — a standing, signed,
idempotent weight-of-NAV target book + ONE causal single-account netted book whose NAV
is the single scored object + a typed fail-closed verdict — genuinely dissolves
signal-stacking, implicit leverage, the fail-open cap, and the two-PnL divergence at the
correct boundary. The data/time→target ; price-path→`RiskRule` line is leakage-safe and
well-argued (D2).

It is **not yet sound** as written because the **legacy delete-list is materially
incomplete** and the **spec deltas under-cover the on-disk specs they modify** — both of
which directly threaten the no-fallback guarantee (PRD NG5/C-5/NFR-NO-LEGACY) and would
force post-cutover patches. The blockers are removal-completeness and spec-coverage, not
the design's core shape. Fix the items below before implementation starts.

## Over-engineering flags (candidates to CUT)

1. **`MarketModel` interface as a first-class deliverable (task 2.1, D8).** For v1 the only term is crypto-perp funding, already correct in `funding.py`. A formal swappable interface now is speculative generalization (PRD: "leave extension points only when justified"; NFR-ROOT-CAUSE warns against "the new code path"). **Recommendation:** keep the book's step-4 as a small, well-named internal seam (a function the walk calls), not a published `MarketModel` abstraction/registry. Add the interface when the *second* term (F4 financing) actually lands — that is the Open/Closed moment, and you'll know the real shape then. The design's own D8 argument ("deferring is not doing it again because the architecture is fixed") does **not** require the interface to exist now; it requires the *book* to localize frictions to one call site. Build the call site; defer the interface. (Not a blocker — but it is the clearest over-build.)

2. **Keeping VBT as a cross-check (D9 + Open Question).** Genuinely valuable *only if* the oracle is generalized to the netted book (Finding #8); as-is it validates `trade_count==1` only. Either commit to generalizing it (then it earns its keep) or retire VBT on the quick-run-adjacent path and lean on the reconciliation test (task 7.5). Leaving it undecided ("defaulting to keep, confirm during implementation") is scope ambiguity in a "must-be-correct-before-build" change. **Recommendation:** decide now.

Everything else is right-sized. The `RiskRule` overlay is **essential, not scope-creep**: D2's argument is correct — a causal strategy cannot self-place a price-path stop without lookahead, so the engine-enforced overlay is the *only* causal expression of a stop. The re-entry latch (D2, spec) is likewise essential (moves the candidate's hand-rolled `active_until_by_symbol` into the engine where it belongs).

## What the proposal got RIGHT (preserve — do not "fix")

- **Root-cause boundary is correct.** The fix is at the decision contract + the accounting spine, not a guard on top — matches NFR-ROOT-CAUSE and the review §1A. Verified: the linear sum (`evaluation.py:108-111`), fail-open (`runner/__init__.py:852`), three money-models, and the conceded single-trade oracle (`agreement.py:94-97`) are all real and all dissolve under the target-book + one-book design.
- **Idempotent standing target as the anti-stacking primitive (D1).** Making re-emission a no-op renders signal-stacking *structurally* inexpressible — a stronger guarantee than any runtime cap. This is the right primitive; do not weaken it to a per-bar snapshot.
- **Per-symbol running-quantity netting requires the quantity-delta model (D3).** Correct that this is *not* a surgical add to the window-replay book — real netting needs the delta model. Skipping the "net inside old book" intermediate (review 0a) is justified given the authorized rewrite.
- **data/time→target ; price-path→`RiskRule` line (D2).** Leakage-safe and the only causal way to express stops. Preserve.
- **Fail-closed, never-clamp verdict (D5).** Clamping would re-free leverage and make `target 1.5` and `target 5.0→clamped 1.5` indistinguishable — exactly the leak. The typed reason is the actionable training signal the consumer's failure-interpretation needs (`program.md`, `score_research.md:148`). Preserve.
- **At-risk-bar statistics + min-sample gate (proposal "What Changes", spec).** Directly fixes the #1 review finding (F2): the flat-bar `effective_sample_size` inflation that collapses SE and drives the climbed PSR→1.0. The book knowing active-vs-flat is the structural enabler. Preserve — this is load-bearing for "passes Train ⟹ tradeable."
- **NAV authoritative, ledger as derived attribution kept first-class for alpha/IC (D4).** Aligns with the consumer contract (`score_research.md` climbs foundation PSR, demotes trade economics to diagnostics) and preserves the "is the signal predictive?" question NAV-only can't answer. Correct.
- **Purity + import wall preserved (D7/D9).** The spine stays pure-Python; no heavy deps on the ~1s Train path. Consistent with the memory note ("foundation offers the foundation, not the climb") and PRD NFR-SIMPLICITY/performance. Preserve.
- **Explicit named follow-ons (F4/F1/F5/F6/F8) instead of silent drops.** Good discipline — they are named so they aren't lost. The deferral of F1/F5 is defensible; F4 is defensible **only with** the Finding #6 fail-closed-on-unfinanced-leverage correction.

## Bottom line

The architecture is correct and dissolves the four named defects at the right boundary.
The gating issues are **completeness, not concept**: the delete-list omits the
engine→executable translation layer (L5/L6), the evaluation-side coupling to the
open-ticket model and the perp-ledger funding-model name (L12-L14), the editable leverage
ceiling (L3-finding/Finding #3), DSR (L20), `promotion_eligible` (L19), and the
single-trade oracle (L21); and the spec deltas under-cover the on-disk foundation/economics
specs they modify (Finding #4, #9). Close the two blockers and the four majors, name the
under-spec items, and this is a sound foundation to build. Cut the premature `MarketModel`
interface to one internal call site and decide the VBT question now.
